## 반드시 읽어볼 내용 TOP 5

### 1. MemGraphRAG: Memory-based Multi-Agent System for Graph RAG

출처: arXiv, KDD 2026 accepted
링크: https://arxiv.org/abs/2606.00610

중요도: 9/10

왜 중요한가: 기존 GraphRAG가 문서 조각 단위로 그래프를 만들면서 전체 코퍼스 관점의 일관성을 잃는 문제를, shared memory 기반 multi-agent graph construction으로 다룬다.

핵심 요약: 에이전트들이 공유 메모리를 통해 전역 문맥을 유지하며 지식 그래프를 만들고, memory-aware hierarchical retrieval을 수행한다. 논문은 여러 벤치마크에서 SOTA baseline 대비 우수하다고 주장한다.

실무 적용 가능성: 문서 수가 늘어날수록 단순 chunk embedding이 깨지는 엔터프라이즈 RAG에서 유용하다. 다만 그래프 구축 비용과 품질 평가 체계가 필요하다.

민수님 기술스택과 관련성: OpenSearch/Elasticsearch의 BM25+vector 후보군 위에 graph expansion layer를 얹거나, Milvus 검색 결과를 entity graph로 확장하는 실험으로 연결하기 좋다.

### 2. Retrieval as Reasoning: Self-Evolving Agent-Native Retrieval via LLM-Wiki

출처: arXiv
링크: https://arxiv.org/abs/2605.25480

중요도: 9/10

왜 중요한가: RAG를 "top-k chunk lookup"이 아니라 agent가 search/read/link-following을 반복하는 reasoning process로 재정의한다.

핵심 요약: 문서를 wiki-like structured pages와 bidirectional links로 컴파일하고, 에이전트가 도구 호출로 탐색한다. HotpotQA, MuSiQue, 2WikiMultiHopQA 등에서 GraphRAG/LightRAG 계열보다 높은 성능을 보고한다.

실무 적용 가능성: 사내 문서, 기술 위키, 장애 보고서처럼 cross-reference가 많은 데이터에 잘 맞다. 단순 벡터 DB보다 ingestion 설계가 더 중요해진다.

민수님 기술스택과 관련성: BigQuery에 문서-엔티티-링크 테이블을 두고, OpenSearch/Milvus는 검색 후보 생성, LLM agent는 link traversal을 담당하게 만들 수 있다.

### 3. SAG: SQL-Retrieval Augmented Generation with Query-Time Local Structure

출처: arXiv
링크: https://arxiv.org/abs/2606.15971

중요도: 8/10

왜 중요한가: 모든 문서에 대해 거대한 전역 그래프를 미리 만들지 않고, query time에 SQL join으로 local hyperedge를 구성하는 접근이다.

핵심 요약: chunk를 semantically complete event와 indexing entities로 변환한 뒤, 질의 시점에 공통 entity를 기준으로 연결 구조를 만든다. GraphRAG의 구축 비용과 최신성 문제를 줄이려는 방향이다.

실무 적용 가능성: 데이터엔지니어에게 특히 좋다. warehouse-backed RAG, SQL 기반 평가, lineage 추적과 잘 맞다.

민수님 기술스택과 관련성: BigQuery로 entity/event table을 만들고, OpenSearch 또는 Milvus 검색 결과를 SQL join으로 확장하는 구조를 실험하기 좋다.

### 4. MASDR-RAG: Multi-Agent Scoped Domain Retrieval for RAG

출처: arXiv
링크: https://arxiv.org/abs/2606.11350

중요도: 8/10

왜 중요한가: 코퍼스가 커지면 hybrid dense+sparse retrieval에서도 정확도가 급락하는 "vector search dilution" 문제를 정면으로 다룬다.

핵심 요약: 실제 도메인 코퍼스에서 문서 수가 54개에서 1,128개로 늘자 정확도가 75%에서 40% 미만으로 떨어졌다고 보고하고, domain-scoped retrieval을 multi-agent 방식으로 제안한다.

실무 적용 가능성: 운영 RAG에서 가장 흔한 문제 중 하나다. index를 하나로 크게 만드는 대신 domain routing, metadata filtering, per-domain retriever를 설계해야 한다.

민수님 기술스택과 관련성: Elasticsearch/OpenSearch index alias, Milvus collection partition, metadata filter, Kafka 기반 도메인별 ingestion routing 실험으로 바로 연결된다.

### 5. SMSR: Certified Defence Against Runtime Memory Poisoning in RAG Agents

출처: arXiv
링크: https://arxiv.org/abs/2606.12703

중요도: 8/10

왜 중요한가: 장기 메모리를 쓰는 RAG agent에서 사용자가 정상 채널로 악성 memory를 주입하고, 나중에 retrieval될 때 응답을 오염시키는 공격면을 다룬다.

핵심 요약: persistent memory RAG agent의 runtime memory poisoning을 방어하는 방법을 제안한다. 에이전트 메모리가 제품 기능으로 들어갈수록 retrieval 품질만큼 보안 평가가 중요해진다.

실무 적용 가능성: memory store, vector DB, tool logs를 운영하는 서비스라면 "누가 쓴 memory가 어떤 사용자에게 검색되는가"를 반드시 통제해야 한다.

민수님 기술스택과 관련성: Milvus/OpenSearch에 user_id, tenant_id, trust_score, source_type metadata를 넣고 retrieval filter와 audit log를 설계하는 프로젝트로 만들 수 있다.

## 취업 관점 분석

지금 시장에서 가장 팔리는 조합은 "RAG를 만들 줄 안다"가 아니라 RAG를 평가하고 고칠 줄 안다는 것이다. 특히 hybrid search, reranking, metadata filtering, query decomposition, GraphRAG, retrieval metrics를 실제로 비교한 결과물이 중요하다.

스택 조합은 OpenSearch/Elasticsearch + Milvus + Kafka + BigQuery가 좋다. Kafka로 문서 ingestion과 re-indexing 이벤트를 관리하고, OpenSearch는 sparse/hybrid 검색, Milvus는 dense vector, BigQuery는 평가 로그와 offline analytics를 맡기는 구조가 실무적으로 설득력 있다.

민수님 이력서에는 "정확도 몇 % 개선"보다도 nDCG@k, Recall@k, MRR, answer faithfulness, latency, cost까지 같이 측정한 표가 들어가면 좋다. 특히 코퍼스 크기가 커질 때 품질이 어떻게 무너지는지 보여주고 개선하면 Retrieval Engineer 포지션에 잘 맞다.
