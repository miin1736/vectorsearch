## 반드시 읽어볼 내용 TOP 5

### 1. CompRank: Efficient LLM Reranking via Token-Level Compression and Decoding-Free Scoring

출처: arXiv  
링크: https://arxiv.org/html/2606.11700v1

중요도: 9/10

왜 중요할까:

LLM reranker는 RAG 품질을 올리는 데 효과적이지만 top-100 이상 후보를 다루면 attention 비용과 latency가 병목이 된다. CompRank는 문서 토큰을 압축하고 autoregressive decoding 없이 attention 기반 점수로 reranking하는 방향을 제시한다. 논문은 7개 BEIR 데이터셋에서 전체 토큰 attention 대비 평균 NDCG@10 손실을 작게 유지하면서 문서 토큰을 10.2%만 남겼다고 보고한다.

스카우트 요약:

핵심은 reranker를 "좋은 모델을 더 세게 돌리는 단계"가 아니라 "ranking signal이 있는 토큰만 남기는 계산 그래프"로 보는 것이다. full-token LLM reranker와 generation-based listwise reranker 사이에서 비용-품질 타협점을 만들 수 있다.

실무 적용 가능성:

OpenSearch/Elasticsearch 또는 Milvus에서 first-stage 후보를 100-500개 뽑은 뒤, reranker 입력 토큰 수와 latency를 줄이는 실험으로 바로 연결할 수 있다. BigQuery에는 query, candidate count, input token count, rerank latency, nDCG/Recall을 저장해 Pareto frontier를 만들면 좋다.

민수님의 기술스택과 관련성:

OpenSearch, Elasticsearch, Milvus, BigQuery, RAG. 검색 엔지니어 면접에서 "reranker를 붙였다"보다 "reranker 비용을 측정하고 줄였다"가 더 강한 증거가 된다.

### 2. Tail-Aware Adaptive-k: Query-Adaptive Context Selection for Retrieval-Augmented Generation

출처: arXiv  
링크: https://arxiv.org/html/2606.11907v1

중요도: 8.5/10

왜 중요할까:

대부분의 RAG 시스템은 top-k를 고정하지만, 실제 query마다 유효한 context 개수는 크게 다르다. 이 논문은 ranked similarity score의 곡선에서 relevance-to-noise 전환점을 찾아 query별로 context를 잘라내는 training-free adaptive-k 방법을 제안한다.

스카우트 요약:

방법은 knee detection으로 후보 구간을 좁힌 뒤, EVT-inspired tail validation으로 noise tail이 안정되는 지점을 찾는다. 논문은 WebQuestions, 2WikiMultiHopQA, MuSiQue에서 oracle truncation에 가까운 retrieval quality를 더 낮은 계산비용으로 얻었다고 보고한다.

실무 적용 가능성:

Milvus나 Elasticsearch kNN 결과의 score distribution을 저장해 query별 adaptive-k를 계산해볼 수 있다. 고정 top-5/top-10/top-20과 비교해서 answer accuracy, hallucination rate, token cost, latency를 함께 측정하면 실무형 RAG 평가 자료가 된다.

민수님의 기술스택과 관련성:

Milvus, OpenSearch, Elasticsearch, BigQuery, Gemini Embedding, RAG. BigQuery에 score curve와 answer judge 결과를 쌓으면 운영 로그 기반 adaptive-k 정책으로 확장할 수 있다.

### 3. uva-irlab-conv at SemEval-2026 Task 8: Multi-Turn RAG with Learned Sparse Retrieval and Listwise Reranking

출처: arXiv / SemEval-2026  
링크: https://arxiv.org/abs/2606.11945

중요도: 8/10

왜 중요할까:

멀티턴 RAG는 단발 QA보다 어렵다. 이전 대화 맥락을 query rewriting, retrieval, reranking, answer generation에 어떻게 넣을지가 품질을 좌우한다. 이 시스템은 finance, cloud documentation, government, Wikipedia 네 도메인과 unanswerable query까지 포함한 평가에서 learned sparse retrieval과 LLM 기반 pointwise/listwise reranking을 조합한다.

스카우트 요약:

흥미로운 점은 dense retrieval을 기본값으로 두지 않고 learned sparse retrieval을 1차 검색으로 둔 것이다. 도메인이 바뀌는 conversational RAG에서는 lexical/sparse generalization이 여전히 강한 선택지라는 신호다.

실무 적용 가능성:

OpenSearch/Elasticsearch의 BM25 또는 learned sparse 계열을 primary retriever로 두고, Milvus dense result를 보조 후보로 fusion한 뒤, listwise reranking을 대화 이력 조건으로 평가할 수 있다. unanswerable query를 평가셋에 넣는 것도 중요하다.

민수님의 기술스택과 관련성:

OpenSearch, Elasticsearch, Milvus, BigQuery, RAG. 고객지원/문서 QA형 RAG에서 multi-turn query rewriting, evidence sufficiency, abstention을 평가하는 프로젝트로 좋다.

### 4. Energy-Efficient On-Device RAG on a Mobile NPU

출처: arXiv  
링크: https://arxiv.org/html/2606.11257v1

중요도: 7.5/10

왜 중요할까:

RAG는 embedding, retrieval, reranking, generation이 모두 들어가서 edge/on-device 배포가 어렵다. 이 논문은 Snapdragon X Elite의 Hexagon NPU에서 embedding, reranking, LLM generation을 포함한 end-to-end RAG를 실행하고 CPU/GPU baseline과 에너지 및 latency를 비교한다.

스카우트 요약:

논문은 NPU가 indexing에서 embedding throughput을 크게 높이고, query workload에서도 CPU 대비 latency와 energy를 줄였다고 보고한다. 품질 평가는 LLM-as-judge 기준으로 backend 간 큰 차이가 없었다고 한다. 서버 RAG와는 다른 축인 "전력당 RAG 성능"이 중요해지는 흐름이다.

실무 적용 가능성:

민수님 장비는 NPU 중심은 아니지만, local RAG에서 CPU/GPU별 embedding batch size, reranker quantization, generation latency를 측정하는 실험으로 바꿔볼 수 있다. 16GB VRAM 환경에서는 작은 reranker와 embedding 모델의 throughput 측정이 현실적이다.

민수님의 기술스택과 관련성:

RAG, Gemini Embedding, Milvus, BigQuery. 온디바이스 자체보다 "RAG pipeline stage별 latency/energy/cost profiling" 관점이 채용 시장에서 실무적으로 더 쓸모 있다.

### 5. Findings of the MAGMaR 2026 Shared Task

출처: arXiv / ACL MAGMaR 2026 Shared Task  
링크: https://arxiv.org/html/2606.12295v1

중요도: 7.5/10

왜 중요할까:

멀티모달 augmented generation에서 video retrieval과 grounded article generation을 다룬 shared task 결과다. 110,000개 이상 multilingual, event-centric video collection을 대상으로 retrieval과 generation을 평가했다. 단일 dense embedding보다 넓은 1차 recall과 reasoning-based reranking이 강했다는 점이 RAG 실무와 직접 맞닿아 있다.

스카우트 요약:

가장 실무적인 관찰은 "비디오를 직접 이해하기보다 caption/summary로 텍스트화한 뒤 reasoning하는 방식이 강했다"는 점이다. 또한 evidence가 부족할 때 abstain하는 시스템이 사람 평가에서는 보상받지만 자동 metric에서는 손해를 볼 수 있다는 평가 설계 문제가 드러났다.

실무 적용 가능성:

이미지/PDF/영상 RAG를 만들 때 원본 modality만 밀어붙이기보다 OCR, caption, summary, metadata를 별도 index로 만들어 hybrid retrieval을 구성하는 쪽이 실용적이다. OpenSearch에는 text/caption/metadata를, Milvus에는 embedding을, BigQuery에는 modality별 retrieval contribution을 저장해 분석할 수 있다.

민수님의 기술스택과 관련성:

OpenSearch, Elasticsearch, Milvus, BigQuery, RAG. 멀티모달 RAG 프로젝트를 "caption index + dense vector + reranker + evidence sufficiency 평가"로 구조화할 수 있다.

## 취업 관련 분석

현재 AI Data Engineer / Retrieval Engineer 포지션에서 가장 시장성 있는 조합은 hybrid search, reranking, retrieval evaluation, observability다. OpenSearch/Elasticsearch를 lexical/hybrid 검색 엔진으로 다루고, Milvus 같은 vector DB를 dense candidate 생성에 붙이며, BigQuery로 offline evaluation과 운영 로그 분석을 하는 구조가 강하다.

Agent 쪽은 MCP나 multi-agent라는 이름 자체보다 tool-call log, state management, evidence verification, budget control을 보여주는 프로젝트가 더 설득력 있다. RAG 쪽도 단순 데모보다 answerable/unanswerable 평가셋, adaptive-k, reranker cost profiling, query decomposition 위치 실험처럼 수치가 남는 작업이 이력서에 강하다.

무시해도 되는 신호는 "벡터 DB 하나로 RAG 해결", "agent framework 비교만 하는 글", "제품 출시/투자 뉴스"다. 오늘 자료 기준으로는 dense retrieval 단독보다 sparse/hybrid + reranking + 평가 설계가 더 실무 가치가 높다.
