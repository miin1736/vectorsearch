## 반드시 읽어볼 내용 TOP 5

### 1. GraphER: Efficient Graph-Based Enrichment and Reranking for RAG

출처: arXiv, Oracle AI
링크: https://arxiv.org/html/2603.24925v2

중요도: 9/10

왜 중요한가: 일반 semantic search가 multi-hop, table retrieval, long-document QA에서 필요한 증거를 일부 놓치는 문제를 "쿼리 시점 그래프 구성 + graph-based reranking"으로 보완한다. 별도 지식그래프 인프라를 두지 않고 기존 vector store 위에 붙일 수 있다는 점이 실무적으로 크다.

핵심 요약: 문서 간 구조적, 개념적, 문맥적 proximity를 후보 문서 그래프로 만들고, 이 관계를 reranking에 반영한다. 논문은 table retrieval, multi-hop retrieval, long-document retrieval에서 retrieval completeness 개선을 보고한다.

실무 적용 가능성: OpenSearch/Elasticsearch의 BM25 + vector hybrid 후보군을 넓게 뽑은 뒤, metadata 관계, 문서 계층, 링크, foreign key, parent-child 구조를 graph reranker feature로 넣는 식으로 구현해볼 수 있다. Milvus만 쓰는 경우에도 vector 후보군 이후 별도 rerank layer로 붙이기 좋다.

민수님 기술스택과 관련성: RAG, OpenSearch, Elasticsearch, Milvus, BigQuery 평가 로그와 직접 연결된다. 특히 테이블/문서/엔티티 관계가 있는 사내 지식검색에 잘 맞는다.

### 2. ANN Search: Recall What Matters

출처: arXiv
링크: https://arxiv.org/html/2606.04522v1

중요도: 8.5/10

왜 중요한가: vector DB 튜닝에서 Recall@k만 높이는 것이 실제 RAG 품질 대비 과한 비용을 만들 수 있다는 문제를 정면으로 다룬다. 검색 엔지니어링 관점에서 매우 실용적이다.

핵심 요약: 저자들은 ANN 평가에서 identifier overlap 기반 Recall@k보다 거리 품질을 보는 1/Ratio@k가 downstream RAG 품질과 더 잘 맞을 수 있다고 주장한다. HNSW, Annoy 등 ANN 알고리즘을 비교하며 높은 Recall@k를 강제하는 튜닝이 query time, distance computation, memory, build time 비용을 키울 수 있음을 보인다.

실무 적용 가능성: Milvus/HNSW, Elasticsearch/OpenSearch kNN 튜닝에서 `efSearch`, `M`, candidate size를 무조건 올리기보다 answer quality, semantic similarity, latency, cost를 함께 보는 벤치마크를 만들 근거가 된다.

민수님 기술스택과 관련성: Milvus와 OpenSearch vector search 튜닝, BigQuery 기반 retrieval metric 분석, Kafka로 검색 로그를 수집하는 파이프라인 설계에 바로 연결된다.

### 3. Cost-Aware Query Routing in RAG

출처: arXiv
링크: https://arxiv.org/html/2606.02581v1

중요도: 8/10

왜 중요한가: 모든 질문에 같은 top-k, 같은 reranker, 같은 generation profile을 쓰는 RAG는 비용과 latency를 낭비한다. 실서비스 RAG에서는 이 문제가 채용 면접에서도 자주 나오는 설계 포인트다.

핵심 요약: CA-RAG는 query별로 retrieval-free, shallow retrieval, deeper retrieval 같은 strategy bundle을 선택한다. 논문은 always-heavy retrieval 대비 billed token 26% 감소, always-direct 대비 평균 latency 34% 감소를 보고하면서 answer quality는 유지했다고 설명한다.

실무 적용 가능성: query complexity classifier를 두고 단순 FAQ는 BM25/top-k 소량, 복잡 질의는 hybrid + reranker + 더 큰 context로 라우팅하는 구조를 만들 수 있다. 운영 로그를 BigQuery에 쌓으면 비용-품질 Pareto 분석까지 가능하다.

민수님 기술스택과 관련성: Kafka로 query/request 로그 수집, BigQuery로 비용/latency/quality 분석, OpenSearch/Elasticsearch retrieval depth 조정에 유용하다.

### 4. Lost in the Evidence? Reproducing Document Position and Context Size Effects in RAG

출처: arXiv / SIGIR 2026
링크: https://arxiv.org/html/2605.27105v1

중요도: 8/10

왜 중요한가: "관련 문서를 앞에 둘까, 원문 순서를 유지할까, 몇 개까지 넣을까" 같은 RAG context 구성 이슈는 실무에서 답이 애매하다. 이 논문은 position bias와 context size 효과를 재현성 관점에서 다시 본다.

핵심 요약: 작은 topic set은 ordering effect를 과장하거나 숨길 수 있고, retrieval quality와 model choice에 따라 context order/context size 효과가 강하게 달라진다고 분석한다. 이상적인 oracle retrieval 결과가 실제 retriever 기반 RAG로 그대로 이전되지 않는다는 점도 중요하다.

실무 적용 가능성: "top-k 늘리면 좋아진다"가 아니라 topic 수, 질의 유형, retriever 품질, LLM judge 편향을 통제한 평가셋을 먼저 만들어야 한다. RAG 평가 노트북이나 BigQuery dashboard 주제로 좋다.

민수님 기술스택과 관련성: OpenSearch/Milvus retrieval 결과의 ordering 실험, Gemini Embedding 기반 chunk 순서 실험, BigQuery 평가 테이블 설계에 바로 연결된다.

### 5. Chunking Methods on Retrieval-Augmented Generation

출처: arXiv
링크: https://arxiv.org/html/2606.00881v1

중요도: 7.5/10

왜 중요한가: chunking은 너무 흔해서 가볍게 취급되지만, 실제 RAG 품질과 비용을 크게 흔든다. 이 논문은 다양한 chunking 방법의 효과, 비용, 한계를 비교하려는 흐름이라 실무자에게 학습 가치가 크다.

핵심 요약: fixed-size와 semantic chunking을 포함해 여러 chunking 전략이 데이터 유형과 use case에 따라 다르게 동작하며, 작은 chunk는 정보 부족, 큰 chunk는 noise 증가 문제를 만든다고 정리한다.

실무 적용 가능성: 문서 타입별 chunk policy를 분리하고, chunk size/overlap/semantic splitting을 retrieval metric과 answer metric으로 같이 비교하는 실험을 만들 수 있다.

민수님 기술스택과 관련성: Gemini Embedding, OpenSearch/Elasticsearch indexing strategy, Milvus collection 설계, BigQuery 기반 offline evaluation과 관련이 크다.

- 예상 소요시간: 4-6시간

- 이력서 활용 가능성: vector DB 성능 튜닝과 RAG 품질 평가를 함께 보여주는 포트폴리오가 된다.

3. Query complexity 기반 Cost-Aware RAG router
- 예상 소요시간: 8-10시간

- 기대 효과: 단순 질의는 shallow retrieval, 복잡 질의는 hybrid + reranker로 보내 token/latency 절감률과 품질 유지 여부 측정

## 취업 관련 분석

현재 시장성은 "RAG를 써봤다"보다 "검색 품질을 측정하고 비용/latency/정확도를 튜닝할 수 있다"가 훨씬 강하다. 특히 OpenSearch/Elasticsearch hybrid search, Milvus/HNSW 튜닝, reranker integration, BigQuery 기반 offline evaluation 조합은 AI 데이터엔지니어와 Retrieval Engineer 양쪽에 모두 유효하다.

가장 어필되는 스택 조합은 `Kafka 로그 수집 -> BigQuery 평가/분석 -> OpenSearch 또는 Milvus 검색 실험 -> RAG quality dashboard`다. 여기에 chunking, reranking, query routing 중 하나를 명확히 개선한 실험 결과가 있으면 단순 토이 RAG보다 훨씬 설득력이 있다.

무시해도 되는 것은 기술 세부가 없는 agent/MCP 큐레이션 리스트, 단순 스타 수 랭킹, 투자/홍보성 product launch다. 오늘 기준으로는 agent framework를 하나 더 따라가기보다 retrieval evaluation과 운영 비용 최적화 쪽이 민수님 커리어에 더 직접적이다.
