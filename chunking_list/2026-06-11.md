## 반드시 읽어야 할 내용 TOP 5

### 1. EviProp: Seeded Relevance Diffusion on Chunk-Page Graphs for Long Multimodal Document Retrieval

출처: arXiv, East China Normal University / Shanghai AI Laboratory  
링크: https://arxiv.org/html/2606.08979v1

중요도: 9/10

왜 중요할까: 긴 PDF, 리포트, 매뉴얼, 금융 문서 RAG에서 "정답 증거가 있는 페이지"를 찾는 문제는 단순 page-level embedding으로 잘 풀리지 않는다. EviProp은 문서를 page node와 chunk node가 섞인 그래프로 만들고, dense visual page prior와 sparse chunk seed를 섞은 뒤 Personalized PageRank로 relevance를 확산한다.

핵심 요약: 저자들은 visually rich long document에서 각 페이지를 독립적으로 점수화하면 작은 표 셀, 숫자, 차트 영역 같은 국소 증거가 묻힌다고 본다. 그래서 offline 단계에서 Chunk-Page graph를 만들고, online 단계에서 query 기반 seed를 넣어 evidence page를 재랭킹한다. MMLongBench-Doc, LongDocURL에서 visual retrieval 및 text-visual fusion baseline 대비 일관된 개선을 보고했다.

실무 적용 가능성: 문서 RAG에서 "chunk 검색 후 parent page/doc 재랭킹"을 더 구조적으로 만들 수 있다. OpenSearch/Elasticsearch에는 sparse lexical seed와 metadata edge를, Milvus에는 dense page/chunk embedding을, BigQuery에는 page graph와 retrieval log를 저장해 실험하기 좋다.

민수님 기술스택과 관련성: RAG, OpenSearch, Elasticsearch, Milvus, BigQuery, Gemini Embedding. 특히 PDF 기반 사내 문서 검색, 표/차트 포함 리포트 QA, evidence page recall 평가 프로젝트로 이력서화하기 좋다.

### 2. ColBERTSaR: Sparsified ColBERT Index via Product Quantization

출처: arXiv / SIGIR 2026  
링크: https://arxiv.org/html/2606.05568v1

중요도: 8.5/10

왜 중요할까: ColBERT류 late-interaction retrieval은 품질은 좋지만 인덱스 크기와 query-time gathering/decompression 비용이 부담이다. 이 논문은 ColBERT embedding quantization을 learned sparse retrieval과 연결해 보고, PLAID one-bit index보다 50-70% 작은 인덱스로 검색 효과를 유지한다고 보고한다.

핵심 요약: multi-vector retrieval은 document token 단위 표현을 보존해 single-vector dense retrieval보다 세밀하지만 운영 비용이 크다. ColBERTSaR는 product quantization을 이용해 ColBERT 인덱스를 더 진짜 inverted-index에 가까운 형태로 바꾸려는 접근이다.

실무 적용 가능성: 당장 production에 넣기보다는 "BM25 + dense + late-interaction rerank"를 비교하는 실험에 적합하다. OpenSearch/Elasticsearch의 sparse 검색과 Milvus dense 검색을 붙이고, ColBERT 계열 reranker를 offline benchmark로 돌려 cost-quality frontier를 만들 수 있다.

민수님 기술스택과 관련성: OpenSearch, Elasticsearch, Milvus, RAG. 검색 엔지니어링 면접에서 "왜 late interaction이 좋은지, 왜 운영 비용이 문제인지, 어떻게 압축/후보축소로 완화하는지"를 설명할 좋은 소재다.

### 3. Evaluating RAG Reliability under Clean, Misleading, and Mixed Retrieval

출처: arXiv / ROMCIR 2026  
링크: https://arxiv.org/html/2606.07783v1

중요도: 8/10

왜 중요할까: RAG는 retrieved context가 신뢰 가능하다는 암묵적 가정 위에서 돌아가지만, 실제 검색 결과에는 오래된 정보, 잘못된 정보, 그럴듯한 오정보가 섞인다. 이 논문은 clean, poisoned, mixed evidence 조건에서 LLM이 기존 parametric knowledge와 retrieved evidence 사이에서 어떻게 흔들리는지 평가하는 프로토콜을 제안한다.

핵심 요약: 모델이 retrieval 없이도 정답을 맞히는 factoid question을 대상으로, 올바른 context, 틀린 context, 혼합 context를 주입한다. 그 뒤 parametric override와 confidence inflation 같은 관점에서 RAG가 오정보에 얼마나 취약한지 본다.

실무 적용 가능성: RAG 평가셋을 만들 때 "정답 문서만 있는 happy path"를 넘어서 잘못된 chunk, 오래된 문서, 충돌 문서를 의도적으로 섞는 테스트가 필요하다. BigQuery에 query, retrieved docs, answer, judge result, conflict label을 쌓으면 운영 품질 모니터링으로 확장 가능하다.

민수님 기술스택과 관련성: RAG, BigQuery, OpenSearch, Elasticsearch. 검색 품질뿐 아니라 "신뢰성 평가 설계"를 보여주는 프로젝트 증거로 가치가 높다.

### 4. Harness-1: Reinforcement Learning for Search Agents with State-Externalizing Harnesses

출처: Hugging Face Papers, Chroma  
링크: https://huggingface.co/papers/2606.02373

중요도: 8/10

왜 중요할까: 검색 agent가 긴 transcript 안에서 모든 상태를 기억하게 만들면 후보 문서, 검증 상태, evidence link, budget 관리가 context를 잡아먹는다. Harness-1은 semantic decision은 모델이 하되, candidate pool, curated evidence, verification records, budget-aware context rendering은 harness가 관리하는 구조를 제안한다.

핵심 요약: 20B search agent를 stateful search harness 안에서 RL로 학습했다. 8개 retrieval benchmark에서 average curated recall 0.730을 기록하고, 다음 강한 open search subagent보다 +11.4 point 높았다고 보고한다. 핵심 메시지는 "모델만 학습 대상이 아니라 harness 설계도 검색 성능의 일부"라는 점이다.

실무 적용 가능성: Agentic RAG를 만들 때 conversation history에 모든 것을 누적하지 말고, 검색 상태를 구조화된 table/state store로 빼는 설계가 중요하다. Kafka로 search event를 수집하고 BigQuery에 candidate/evidence/verification 테이블을 두면 재현 가능한 agent retrieval harness를 만들 수 있다.

민수님 기술스택과 관련성: RAG, Kafka, BigQuery, OpenSearch, Elasticsearch, Milvus. agent 경력 방향까지 고려하면 검색 상태 외부화는 꽤 시장성 있는 주제다.

### 5. miniReranker: Efficient Multimodal Reranking through Visual Cache Reuse and Interaction Sparsity

출처: arXiv  
링크: https://arxiv.org/html/2606.10759v1

중요도: 7.5/10

왜 중요할까: 멀티모달 RAG에서 MLLM reranker는 품질은 좋지만 후보마다 query-document pair를 다시 계산해야 해서 비용이 크다. miniReranker는 vision-first prompt, prefix caching, early exit, interaction band, visual token pruning으로 reranking 비용을 줄인다.

핵심 요약: Qwen3-VL 기반으로 active parameter를 58% 수준으로 줄이고, top-k 후보 reranking runtime을 최대 99% 줄일 수 있다고 보고한다. 수치 자체는 재현 검증이 필요하지만, "비싼 multimodal reranker를 어떻게 production-friendly하게 만들 것인가"라는 문제의식은 실무적으로 중요하다.

실무 적용 가능성: 문서 이미지 검색에서 1차 retrieval은 OpenSearch/Milvus로 넓게 가져오고, 2차 multimodal reranker는 candidate 수, cache reuse, early-exit 정책을 달리해 비용-정확도 곡선을 만들 수 있다.

민수님 기술스택과 관련성: RAG, Milvus, OpenSearch, Elasticsearch, BigQuery. PDF/이미지 문서 검색의 reranker benchmark 프로젝트로 이어질 수 있다.

## 취업 관련 분석

현재 AI Data Engineer / Retrieval Engineer 시장에서 가장 강한 조합은 "RAG를 만들 줄 안다"가 아니라 "검색 품질을 측정하고 운영 비용을 제어할 줄 안다"에 가깝다. OpenSearch/Elasticsearch 기반 lexical/hybrid retrieval, Milvus 같은 vector DB, reranker, BigQuery 기반 평가/로그 분석을 묶을 수 있으면 실무형 포트폴리오가 된다.

가장 시장성 있는 스킬은 retrieval evaluation, hybrid search tuning, chunking/indexing strategy, reranker integration, observability다. Agent 쪽은 MCP나 multi-agent 자체보다 state management, tool-call log, evidence verification, budget control처럼 운영 가능한 설계를 보여주는 편이 낫다.

이력서 증거로는 "RAG 챗봇 구현"보다 benchmark table, evaluation notebook, indexing migration note, failure analysis report, cost-quality dashboard가 훨씬 강하다. 특히 BigQuery에 검색 로그와 judge 결과를 쌓아 offline evaluation을 자동화했다는 식의 산출물이 좋다.

무시해도 되는 것은 단순 agent framework 스타 수, 모델 데모 영상, 구체 benchmark 없는 product launch다. 최근 GitHub trending에서는 coding agent와 MCP server가 계속 강하지만, 민수님에게 직접 도움이 되는 것은 repo 자체보다 그 안의 검색/상태관리/평가 패턴이다.
