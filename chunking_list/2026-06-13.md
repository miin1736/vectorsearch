### 1. Topic Is Not Agenda: A Citation-Community Audit of Text Embeddings

출처: arXiv  
링크: https://arxiv.org/abs/2605.07158

중요도: 9/10

왜 중요한가:

이 논문은 vector search와 RAG가 자주 전제하는 "embedding cosine similarity는 개념적 관련성을 잘 잡는다"는 가정을 과학 논문 검색에서 직접 감사한다. 358만 편 규모의 citation graph를 만들고 sub-field(L1)와 research-agenda(L2) 단위로 나눠 Gemini, Qwen3, SPECTER2 계열 embedding을 평가했다. 출처 사실 기준으로, L1 top-10 same-rate는 45-52% 수준이지만 L2에서는 15-21%로 떨어지고, top-10 중 8개가 agenda 관점에서 빗나가는 실패가 관찰된다.

스카우트 요약:

실무 RAG에서 "semantic similarity가 높으니 충분히 관련 있다"는 판단은 위험하다. 특히 논문, 법률, 의료, 사내 기술 문서처럼 같은 주제 안에서도 세부 목적이 갈리는 도메인에서는 dense-only retrieval이 정답 주변의 그럴듯한 오답을 많이 가져올 수 있다. 논문은 단순 citation-count rerank가 일부 조건에서 Gemini cosine retrieval보다 agenda matching에 더 강한 신호를 준다는 점도 보여준다.

실무 적용 가능성:

OpenSearch/Elasticsearch의 BM25 또는 boolean retrieval, Milvus dense retrieval, graph/citation/co-reference signal을 결합한 hybrid retrieval benchmark를 만들기 좋다. BigQuery에는 query, retrieved_doc, citation/community label, answer correctness를 쌓고 Recall@k, nDCG@k, agenda-hit-rate를 비교하면 포트폴리오 증거가 된다.

민수님의 기술스택과 관련성:

OpenSearch, Elasticsearch, Milvus, BigQuery, Gemini Embedding, RAG. "embedding 모델 교체"보다 "검색 실패 유형을 계측하고 rerank/fusion으로 보정했다"는 이야기가 면접에서 더 강하다.

### 2. When Does Mixing Help? Analyzing Query Embedding Interpolation in Multilingual Dense Retrieval

출처: arXiv / ACL 2026 Main Oral  
링크: https://arxiv.org/abs/2606.13537

중요도: 8/10

왜 중요한가:

2026-06-11 제출된 최신 논문으로, 다국어 dense retrieval에서 query embedding interpolation이 언제 도움이 되는지 분석한다. arXiv recent listing 기준 ACL 2026 Main Oral로 표시되어 있어, 단순 트릭성 아이디어보다 검증된 연구로 볼 만하다. 다국어/혼합 언어 검색은 한국어-영어 기술 문서 RAG에서 자주 생기는 문제라 실무 연결성이 높다.

스카우트 요약:

민수님 상황에서는 한국어 질문으로 영어 논문/문서/코드를 찾거나, 영어 제품 문서와 한국어 사내 메모를 함께 검색하는 케이스가 많을 수 있다. 이때 query를 한 언어로만 embed하는 대신 원문 query, 번역 query, 확장 query의 embedding을 섞어 검색하는 방식은 실험하기 쉽고 비용도 낮다. 다만 모든 query에 일괄 적용하면 noise가 늘 수 있으므로 query type별 ablation이 필요하다.

실무 적용 가능성:

Gemini Embedding 또는 다른 multilingual embedding으로 `ko_query`, `en_translation`, `query_expansion` embedding을 만들고 interpolation weight를 0.0-1.0 범위에서 sweep한다. Milvus에 동일 corpus를 올리고, OpenSearch BM25와 RRF로 섞은 뒤 Recall@10, MRR, answer faithfulness를 BigQuery에 기록하면 된다.

민수님의 기술스택과 관련성:

Milvus, Gemini Embedding, OpenSearch, Elasticsearch, BigQuery, RAG. 한국어 사용자 질의와 영어 기술 corpus를 잇는 retrieval 튜닝 프로젝트로 좋다.

### 3. More Than Efficiency: Embedding Compression Improves Domain Adaptation in Dense Retrieval

출처: arXiv  
링크: https://arxiv.org/abs/2601.13525

중요도: 8.5/10

왜 중요한가:

2026-06-02 v2로 갱신된 논문이다. 제목처럼 embedding compression을 단순 저장/속도 최적화가 아니라 domain adaptation에도 도움이 되는 방향으로 해석한다. 실무 vector DB에서는 embedding dimension, index memory, latency, recall의 trade-off가 늘 문제라서 바로 적용 가능한 주제다.

스카우트 요약:

대부분의 RAG 프로젝트는 "큰 embedding 모델 + 큰 차원 = 좋다"로 시작하지만, 실제 운영에서는 Milvus/Elasticsearch kNN index 비용, cache hit, ANN latency, update throughput이 병목이 된다. 이 논문은 compression을 비용 절감뿐 아니라 도메인 일반화 관점에서 볼 수 있게 해 준다. 출처 사실은 논문 제목과 submission history까지이며, 실무적 해석은 "차원 축소/압축도 retrieval 품질 실험 대상"이라는 쪽이다.

실무 적용 가능성:

동일 corpus에 대해 원본 embedding, PCA/OPQ/product quantization 또는 모델 내 compression variant를 비교한다. Milvus HNSW/IVF 설정별로 Recall@k, p95 latency, index size, ingestion time을 측정하고 BigQuery에 적재하면 훌륭한 벤치마크 산출물이 된다.

민수님의 기술스택과 관련성:

Milvus, Elasticsearch, BigQuery, Gemini Embedding, RAG. vector DB 운영 감각을 보여주기 좋은 주제다.

### 4. RAGSmith: A Framework for Finding the Optimal Composition of Retrieval-Augmented Generation Methods Across Datasets

출처: arXiv / Hugging Face Papers  
링크: https://arxiv.org/html/2511.01386v1

중요도: 8/10

왜 중요한가:

RAGSmith는 RAG 품질을 retrieval, reranking, augmentation, prompting, generation 각각의 단일 모듈 최적화로 보지 않고 end-to-end architecture search 문제로 둔다. 출처 기준으로 9개 technique family와 46,080개 feasible pipeline configuration을 탐색 대상으로 정의한다.

스카우트 요약:

실무 RAG 품질 저하는 대개 한 모듈의 문제가 아니라 chunking, query expansion, retriever, reranker, passage filtering, compression, prompt template이 서로 간섭해서 생긴다. RAGSmith의 핵심 학습 포인트는 "좋아 보이는 컴포넌트를 붙이는 것"보다 "데이터셋별 실험 설계와 평가 자동화"가 중요하다는 점이다.

실무 적용 가능성:

OpenSearch BM25, Milvus dense retrieval, hybrid RRF, reranker 유무, chunk size, top-k, context compression 조합을 작은 search space로 줄여 실험한다. BigQuery에 config hash, metric, latency, token cost, hallucination judge score를 저장하고 최적 조합을 찾는 notebook/dashboard를 만들면 실무형 프로젝트가 된다.

민수님의 기술스택과 관련성:

OpenSearch, Elasticsearch, Milvus, BigQuery, RAG. "RAG를 만들었다"보다 "RAG 실험 플랫폼을 만들고 조합별 Pareto frontier를 냈다"가 훨씬 강한 이력서 증거다.

### 5. Data Agents: Levels, State of the Art, and Open Problems

출처: arXiv / SIGMOD 2026 tutorial / GitHub paper list  
링크: https://arxiv.org/abs/2602.04261  
보조 링크: https://github.com/HKUSTDial/awesome-data-agents

중요도: 8/10

왜 중요한가:

GitHub companion repo는 2026-06-03 SIGMOD 2026에서 3시간짜리 Data Agents tutorial이 발표되었다고 공지했다. 논문은 data agent를 L0-L5 autonomy taxonomy로 나누며, 단순 query responder와 완전 자율 data scientist를 혼동하는 문제를 짚는다.

스카우트 요약:

AI Data Engineer 관점에서는 agent hype를 그대로 따라가기보다 "데이터 관리/정제/분석 workflow에서 agent가 어느 수준까지 책임질 수 있는가"를 구분하는 능력이 중요하다. 현재 실무적으로는 L1-L2, 즉 추천/보조/환경 인식/도구 호출 수준이 현실적이고, Proto-L3는 감독 하 orchestration으로 보는 게 맞다.

실무 적용 가능성:

Kafka로 들어오는 문서/로그 이벤트를 BigQuery에 적재하고, agent가 schema profiling, 품질 경고, retrieval evaluation query 생성, dashboard 요약을 수행하게 하는 작은 data-agent prototype을 만들 수 있다. MCP/tool calling은 운영 DB 직접 변경이 아니라 read-only analytics와 evaluation automation부터 시작하는 편이 안전하다.

민수님의 기술스택과 관련성:

Kafka, BigQuery, OpenSearch, Elasticsearch, RAG, AI Agent, MCP. 데이터 엔지니어링과 agent를 연결하는 포트폴리오 주제로 적합하다.

## 취업 관련 분석

현재 시장에서 가장 팔리는 스킬은 단순 LLM 앱 구현보다 retrieval evaluation, hybrid search tuning, reranker integration, vector DB 운영, metric 설계다. 특히 "RAG 품질을 어떻게 측정했는가"를 말할 수 있는 사람이 강하다.

스택 조합은 OpenSearch/Elasticsearch의 lexical retrieval, Milvus 같은 vector DB, BigQuery 기반 평가/로그 분석, Kafka 기반 ingestion을 함께 설명할 수 있으면 좋다. Gemini Embedding 같은 managed embedding은 모델 자체보다 migration/evaluation/비용 관리 관점으로 다루는 게 실무적이다.

이력서 증거로는 benchmark table, reproducible repo, evaluation notebook, latency/quality/cost dashboard, failure analysis 문서가 효과적이다. 반대로 "agent가 모든 걸 자동화한다" 류의 과장된 데모나 benchmark 없는 멀티모달 RAG 소개성 프로젝트는 우선순위를 낮춰도 된다.
